# COMPND.SYSTEMS Domain Readiness

## Purpose

Use this checklist before any project is launched from the COMPND.SYSTEMS hub. The catalog can advertise planned domains, but an app should not set `launchLive: true` until DNS, HTTPS, deployment, and support routing are verified.

## Current Domain Policy

- `https://compnd.systems` remains the public hub.
- Projects with their own formal production domains should link directly from the hub.
- Projects without formal domains can use stable `compnd.systems` subdomains.
- Staging, private, lab, and compliance-sensitive apps remain protected and hidden from public launch CTAs.
- No deploy keys, tokens, Hostinger credentials, or secrets are created as part of this plan.

## Launch Gate

Before setting a catalog entry to `launchLive: true`:

1. Confirm the final production domain.
2. Confirm DNS points to the correct production host.
3. Confirm HTTPS is active on the final domain.
4. Confirm the app deploys from the GitHub source-of-truth branch.
5. Confirm the app has a health or smoke check.
6. Confirm support routing through `/contact?app=<slug>`.
7. Confirm legal, privacy, and compliance notes where relevant.
8. Update `helpers/appCatalog.ts`.
9. Run `pnpm run catalog:validate`.
10. Run `pnpm run sitemap:generate`.
11. Run `pnpm run domain:check` to verify DNS and HTTPS for every `launchLive` app.
12. Run `pnpm run build` and `pnpm smoke:test`.

## Validation Commands

```powershell
pnpm run catalog:validate
pnpm run domain:check
pnpm run domain:check:planned
pnpm run sitemap:generate
pnpm run build
pnpm smoke:test
```

Use `pnpm run domain:check` as the required launch gate before changing an app to `launchLive: true`. It checks only currently live catalog entries and fails on live DNS or HTTPS problems. Use `pnpm run domain:check:planned` as an informational audit of proposed subdomains; planned and gated apps may fail until DNS, hosting, and HTTPS are intentionally configured.
